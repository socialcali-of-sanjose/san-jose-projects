Print view: Social Cali of San Jose — AIO & Local Authority SOP

Purpose

Build durable AI Search, AIO, GEO, AEO, and Local SEO authority for Social Cali of San Jose using a controlled, scalable publishing system that prioritizes truth, trust, and clean entity signals. This SOP is designed to be VA-safe, repeatable, and resistant to index failure, hallucinations, and entity pollution.

Primary objective Publish verifiable, neutral standards content that sustains long-term authority and supports AI-mediated search surfaces.
System property One Topic → One Asset Pair (Tier 0 + Tier 1) with strict link and validation controls.

Core Philosophy (Non-Negotiable)

Accuracy > SpeedIf unsure, stop. Do not guess.
One Topic = One Asset PairTier 0 and Tier 1 must always match.
No Guessing, No InventionMissing input = error.
Trust First, Then ScaleTier 0/Tier 1 must be clean before Tier 2 or volume.
Entity Separation MattersGitHub ≠ Money Site ≠ Google Site.

Roles & Permissions

Trust Engineer / VA (This SOP)

  • Primary focus: Tier 0, Tier 1, GitHub hygiene
  • Systems: GitHub Pages (publish only), Google Sites, indexing tools
  • Forbidden: RB Agent, RSI, bulk posting tools

Part 0 — Entity Foundation (GitHub README)

Purpose: Define the scope, intent, and legitimacy of Social Cali’s GitHub authority layer.

Frequency: One-time setup; quarterly review; event-based updates only (major service change).

Rules

  • No schema / JSON-LD
  • No phone numbers
  • No sales language
  • No guarantees
  • Max 2 outbound links (official site + optional Google Site hub)

Required README sections

  • Repository Overview
  • What This Repository Publishes
  • What This Repository Does Not Publish
  • How This Content Is Intended To Be Used
  • Source & Validation Philosophy
  • About the Publisher

Generation rule: README content is generated using the Canonical README Prompt only.

Part 1 — Daily Workstation Setup

Do not start work until all are open:

  • Dedicated browser profile (Social Cali only)
  • GitHub repository (correct client)
  • Google Sites (correct site)
  • Manager Topic Pack for the day
  • Indexing tool

Rule: Never open two different clients in the same browser window.

Part 2 — Daily Execution Loop (M–F)

This loop is identical every workday.

Step 1 — Pre-Flight Check 10 min

Confirm Manager Topic Pack includes all required fields:

CLIENT_NAME TOPIC_NAME TOPIC_SLUG PUBLISH_DATE (YYYY-MM-DD) VALIDATION_SOURCE_URL (authoritative) MONEY_SITE_ALLOWED (YES/NO) MONEY_SITE_URL (if YES) GITHUB_BASE_URL

Stop condition: If any item is missing, stop and report.

Step 2 — Tier 0: GitHub Service Standard Daily

Rotation schedule

  • Monday: Prompt A
  • Tuesday: Prompt B
  • Wednesday: Prompt C
  • Thursday: Prompt D
  • Friday: Prompt A

Construction rules

  • PAGE_URL = GITHUB_BASE_URL + / + TOPIC_SLUG + .html
  • Output = raw HTML only
  • Outbound links (body) = exactly one link to the validation source

Mandatory QA (all must pass)

  • Paragraph 1 starts exactly with: "[TOPIC_NAME] is defined as..."
  • Footer present: "Last updated: YYYY-MM-DD"
  • One JSON-LD script only
  • @graph contains WebPage + Article
  • IDs end with #webpage and #entity
  • No invented stats, laws, or claims

Publish & index

  • Upload file to GitHub Pages
  • Confirm live URL loads
  • Submit URL to indexer
Step 3 — Tier 1: Google Site Shield Page 25–40 min

Rotation schedule

Same A/B/C/D pattern as Tier 0.

Link rules (text links only)

  • Link #1 → Tier 0 GitHub URL (required)
  • Link #2 → Money site (only if allowed)
  • Max total text links: 2

Mandatory QA

  • Clear “Quick Answer” section
  • Plain-language headings
  • No hype or guarantees
  • No more than 2 text links

Publish & index

  • Publish Google Site page
  • Submit URL to indexer
Step 4 — Daily Logging 10 min

Log the following fields:

  • Topic
  • Tier 0 URL
  • Tier 1 URL
  • Validation source
  • Indexed submitted (Y/N)

Part 3 — Weekly Safety & Audit (Friday)

A) Chain Check

Click through:

  • Tier 1 → Tier 0 → Validation Source
  • Fix any broken link immediately.

B) Kill Switch (Critical)

Search Google:

site:[Tier 0 URL]

If two consecutive Tier 0 pages are not indexed after 14 days: stop new Tier 0 production and notify the manager.

Part 4 — Structured Data Policy (Strict)

Money Site (sanjose.socialcali.com)

Allowed structured data types:

  • WebSite
  • Organization
  • ProfessionalService
  • WebPage
  • BreadcrumbList

GitHub Tier 0 Pages

Allowed:

  • WebPage
  • Article / TechArticle

Forbidden:

  • LocalBusiness
  • ProfessionalService
  • Address / Geo / Phone

Google Site Tier 1 Pages

No structured data of any kind.

GitHub README

No structured data of any kind.

Part 5 — Validation Source Rules

Preferred sources

  • Google Search Central
  • Google Ads Help
  • Meta Business Help
  • SBA.gov
  • SCORE.org
  • AMA.org
  • BBB.org

Forbidden sources

  • Opinion blogs
  • SEO-only blogs
  • Affiliate sites
  • Unverifiable stats pages

Stop condition: If no source is provided, stop and request one.

Part 6 — Global Red Lights (Stop Immediately)

  • AI invents a law, stat, or policy
  • Tier 0 links to a sales page
  • More than allowed links are published
  • Two clients opened in same browser
  • Two consecutive Tier 0 pages fail indexing

Part 7 — Prompt Library (Mandatory — Full Page Generation)

All prompts are designed to generate full, long-form pages (not snippets). Each prompt must output a complete page suitable for publishing.

  • VAs: only edit the input block
  • Never shorten outputs
  • Never summarize
  • Never remove required sections
